# 011 バウンディングボックスIDスイッチ問題 — 要求仕様書

## 1. プロジェクト概要

- **何を作るのか**: 2D姿勢推定出力のトラッキングID問題を定量的に把握し、対策を設計する調査・分析ツール群
- **なぜ作るのか**: 複数人環境での3D三角測量の前提として、同一人物の2Dキーポイントが正しく紐付いている必要がある。現状、person_idが全て-1（トラッキングID未割当）であり、personAssociation()が正しく機能するか検証が必要
- **誰が使うのか**: sakagawa（データ解析者）
- **どこで使うのか**: Linux, Python 3.11, micromamba Pose2Sim環境

## 2. 用語定義

| 用語 | 定義 |
|------|------|
| IDスイッチ | 同一人物のトラッキングIDがフレーム間で変化すること |
| person_id | OpenPose形式JSONの`person_id`フィールド。現状全データで[-1] |
| OpenPose形式JSON | Pose2Simが出力する2D姿勢推定結果のJSON。構造: `{"version": float, "people": [{"person_id": list, "pose_keypoints_2d": list}]}` |
| BB | バウンディングボックス。本調査では「conf > 0.1の有効キーポイントの外接矩形」として算出する |
| IoU | Intersection over Union（2つのBBの重なり度合い。0.0〜1.0） |
| personAssociation | Pose2Simの人物関連付けステップ。複数人環境で同一人物を時間・カメラ間で追跡する |
| ハンガリアン法 | 最小コストマッチングのアルゴリズム。scipy.optimize.linear_sum_assignmentで実装 |

## 3. 機能要求一覧

### FR-011-001: IDスイッチの現状把握（Phase 1）

- **機能名**: IDスイッチ定量分析
- **概要**: 対象データセットの2D姿勢推定結果を解析し、検出人数の変動パターンとフレーム間の人物同一性を定量化する
- **入力**:
  - `Pose2Sim/20251024_osaka-hosp.PortalCam.01hour/pose/cam{01,02,04}_json/` 以下のJSONファイル（約10.8万フレーム×3カメラ）
  - JSONフォーマット: OpenPose形式（`version`, `people[].person_id`, `people[].pose_keypoints_2d`）
- **出力**:
  - カメラごとの検出人数の分布（1人/2人/3人/...）
  - 検出人数が変動したフレームの一覧と変動パターン
  - person_idの値の分布（現状全て-1であることの確認）
  - フレーム間のキーポイント距離によるマッチング距離の分布（カメラごとの平均値、中央値、95パーセンタイル）
  - マッチング失敗（距離が閾値超過）の発生率（カメラごと）
  - 分析結果ドキュメント: `docs/011_id_switch_analysis/phase1_results.md`
  - CSV: `docs/011_id_switch_analysis/test_results/id_switch_events.csv`
- **受け入れ基準**:
  - 全フレームについてperson_idの値が集計されていること
  - 検出人数変動のパターンが分類されていること
  - フレーム間マッチング距離の分布（平均値、中央値、95パーセンタイル）がカメラごとに算出されていること
  - マッチング失敗の発生率がカメラごとに算出されていること
  - 少数フレーム（先頭50フレーム）について手動確認と分析結果が一致することの検証が含まれていること

  - 推奨設定変更案
  - 調査結果ドキュメント: `docs/011_id_switch_analysis/phase2_results.md`
- **受け入れ基準**: ソースコードの該当箇所（ファイル名:行番号）が引用されていること

- 正確な検出人数分布（NaN除外後、全フレーム走査）:
  - cam01: 0人=186(0.2%), 1人=103,570(96.0%), 2人=4,073(3.8%), 3人=8(0.0%)
  - cam02: 0人=802(0.7%), 1人=106,596(98.9%), 2人=429(0.4%), 3人=6(0.0%)
  - cam04: 0人=987(0.9%), 1人=105,585(97.9%), 2人=1,257(1.2%), 3人=4(0.0%)
- Config.toml: `multi_person = false`
- フレームレート: 30fps、解像度: 1920x1080
- 動画（cam01.mp4等）は`videos/`ディレクトリにある。大部分は同一人物1人が映り続けている

## 8. Pose2Simコード調査結果（2026-03-05セッション）

### person_id=-1の原因

`poseEstimation.py:261`で`person_id`は**ハードコードで`[-1]`**が出力される。トラッキング処理（sports2d/deepsort）はkeypoints配列の並び順を整えるが、そのIDはJSON出力に反映されない。

### multi_personの各ステップへの影響

| ステップ | `multi_person=false`（現在） | `multi_person=true` |
|---------|-------------------------------|----------------------|
| poseEstimation（sports2dモード時） | **影響なし** | **影響なし** |
| personAssociation | 再投影誤差最小の**1人を選択**。tracked_keypoint（Neck）で全組み合わせを三角測量し最良を採用 | エピポーラ線距離ベースのアフィニティ行列で複数人対応付け |
| triangulation | `nb_persons_to_detect=1`に強制。people[0]のみ三角測量。**3Dフレーム間追跡なし** | 最大検出人数分を三角測量。sports2dで3Dフレーム間再同定 |

### pose動画のBB IDの正体

pose動画に描画されているIDは、`common.py:1321`の`draw_bounding_box()`で`keypoints`配列のインデックス`i`をそのまま`str(i)`として描画したもの。YOLOの内部IDではなく、`sort_people_sports2d()`による並べ替え後の配列インデックスがIDとなる。

### 同一カメラのJSONからIDスイッチを検出する方法

`sort_people_sports2d()`と同じロジック（キーポイント距離ベース＋ハンガリアン法）をJSONに事後適用すれば、フレーム間の人物対応付けが可能。これはPhase 1の設計（design.md Step 3）そのもの。
