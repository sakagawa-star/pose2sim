# docs/REVIEW_CRITERIA.md

## レビュー基準
以下の観点で要求仕様書と機能設計書をレビューする。

### 1. ドキュメント間の一貫性
- 要求仕様書と機能設計書の間に矛盾がないか
- 用語が統一されているか

### 2. 曖昧性の排除
- 「適切に」「必要に応じて」「など」等の曖昧表現がないか
- すべての条件・閾値・タイムアウト値が具体的数値で記述されているか

### 3. エラーハンドリング・境界条件
- 各機能の異常系（入力不正、接続失敗、タイムアウト等）の振る舞いが定義されているか
- 境界値（0件、1件、上限値）の扱いが明記されているか

### 4. 技術スタック・依存関係
- 使用ライブラリとバージョンが明記されているか
- 対象OS・Python版・実行環境が明記されているか

### 5. データフロー・I/O定義
- モジュール間のデータ型・形式・単位が明記されているか
- ファイル入出力のパス規約・命名規則が定義されているか

### 6. 状態遷移
- GUI/プロセスの状態遷移が網羅的に定義されているか
- 各状態からの遷移条件と遷移先が明示されているか

### 7. 非機能要件
- パフォーマンス目標（FPS、応答時間等）が定義されているか
- ログ出力・設定ファイルの方針が定義されているか

### 8. 詳細設計の十分性
- 実装者が判断に迷う余地がないレベルまで詳細化されているか

### 9. 技術スタックの整合性
- docs/TECH_STACK.md と各設計書の技術スタック記載に矛盾がないか
- 実コードのimportと TECH_STACK.md の記載が一致しているか
- バージョン未固定のライブラリがないか
