# 012 2Dキーポイント推定の暴れ対策 — 要求仕様書

## 1. プロジェクト概要

- **何を作るのか**: 2Dキーポイントの「暴れ」（フレーム間の異常な位置変動）を定量的に検出・分類し、抑制策を設計する調査・分析ツール群
- **なぜ作るのか**: 2Dキーポイントの暴れは3D三角測量の精度を直接劣化させる。原因を特定し、Pose2Simの設定や前処理で対処することで3D精度を改善する
- **誰が使うのか**: sakagawa（データ解析者）
- **どこで使うのか**: Linux, Python 3.11, micromamba Pose2Sim環境

## 2. 用語定義

| 用語 | 定義 |
|------|------|
| 暴れ（jitter） | フレーム間のキーポイント移動量が、そのキーポイントの移動量中央値の5倍を超えること。5.0は探索的な初期値であり、Phase 1の結果に基づいて調整する |
| confidence score | 2D姿勢推定モデルが出力する各キーポイントの信頼度（0.0〜1.0）。RTMposeが未検出キーポイントに割り当てる値は0.0 |
| RTMpose | Pose2Simが使用するTop-down型2D姿勢推定モデル |
| BBパディング | バウンディングボックスを拡大してRTMposeに入力する前処理 |
| BB | conf > 0.1の有効キーポイントの外接矩形として算出する |
| confidence閾値 0.1 | confidence 0.1未満のキーポイントは未検出として扱う。RTMposeの未検出値0.0との境界 |

## 3. 機能要求一覧

### FR-012-001: キーポイント暴れの定量的把握（Phase 1）

- **機能名**: 暴れ検出・分類分析
- **概要**: 各カメラの2Dキーポイント時系列を解析し、暴れの発生頻度・パターン・原因を定量化する
- **入力**:
  - 主作業データ: `Pose2Sim/20260227-dgtw2-lab2/pose/cam{01,02,03,04}_json/`（1,800フレーム×4カメラ、30fps、1920x1080、単一人物）
  - JSONフォーマット: OpenPose形式（`pose_keypoints_2d`: [x, y, confidence] × 26キーポイント）
- **出力**:
  - カメラごと・キーポイントごとの暴れ発生回数
  - 暴れ発生時のconfidence scoreの分布
  - 暴れの原因分類（下記4パターン）とその割合
  - 分析結果ドキュメント: `docs/012_2d_keypoint_jitter/phase1_results.md`
  - 暴れ発生箇所のCSV: `docs/012_2d_keypoint_jitter/test_results/jitter_events.csv`（カメラ名, フレーム番号, キーポイント名, 移動量, confidence, 原因分類）
  - プロット画像: `docs/012_2d_keypoint_jitter/test_results/` に保存
- **受け入れ基準**:
  - 暴れ検出の閾値（移動量中央値の5.0倍）が明示的にパラメータ化されていること
  - 全26キーポイント×全カメラの暴れ発生数が集計されていること
  - 暴れ発生時のconfidence分布が可視化されていること

#### 暴れ原因の分類基準

パターンは排他的に分類する。A > C > D > E の優先順で最初に該当したパターンを採用する。

| パターン | 条件 |
|---------|------|
| A: 画角外 | BBが画面端（x < 10px or x > 1910px or y < 10px or y > 1070px）に接している |
| C: BB小 | BB面積が同一カメラの全フレームBB面積中央値の50%未満 |
| D: 低confidence | 暴れキーポイントのconfidence < 0.3 |
| E: その他 | A, C, Dのいずれにも該当しない |

**注意**: パターンB（遮蔽）は主作業データが単一人物のため該当しない。複数人データでの分析時に追加する。

### FR-012-002: RTMposeモデルとBBパディングの設定調査（Phase 2）

- **機能名**: 姿勢推定設定調査
- **概要**: 現在のRTMposeモデル構成・BBパディング・YOLO検出設定を把握し、改善余地を調査する
- **入力**: Pose2Simソースコード（`poseEstimation.py`）、Config.toml
- **出力**:
  - 使用モデル名・入力解像度・スケルトン定義の一覧
  - BBパディングのコード上の処理フロー
  - YOLO側の設定（検出閾値、NMS閾値、モデル種別）
  - Phase 1の原因別改善提案
  - 調査結果ドキュメント: `docs/012_2d_keypoint_jitter/phase2_results.md`
- **受け入れ基準**: ソースコードの該当箇所（ファイル名:行番号）が引用されていること

### FR-012-003: confidence閾値とフィルタリングの最適化設計（Phase 3）

- **機能名**: フィルタリング戦略設計
- **概要**: Phase 1・2の結果を踏まえ、暴れキーポイントを3D推定に持ち込まないためのフィルタリング戦略を設計する
- **入力**: Phase 1・2の分析結果
- **出力**:
  - confidence閾値を0.1刻み（0.1〜0.9）で変化させた場合の暴れ除去率と正常データ喪失率
  - 推奨パラメータセット（選定基準: 暴れ除去率80%以上かつ正常データ喪失率5%以下を満たす閾値のうち、最もデータ喪失率が低いもの）
  - 推奨パラメータの3D精度への影響をtrc_evaluateで検証する計画
  - 設計ドキュメント: `docs/012_2d_keypoint_jitter/phase3_design.md`
- **受け入れ基準**: 閾値と暴れ除去率/正常データ喪失率のトレードオフが定量的に示されていること

## 4. 非機能要求

- Phase 1分析スクリプトの実行時間: 主作業データ（1,800フレーム×4カメラ = 7,200フレーム）は1分以内（ベストエフォート）
- メモリ使用量: 主作業データの全キーポイントデータ（7,200フレーム × 26 × 3 × 8bytes ≈ 4.3MB）はメモリに保持してよい。2パス処理（1パス目: 移動量・BB面積収集、2パス目: 閾値確定後に暴れ検出）を採用する

## 5. 制約条件

- 主対象データセット: `20260227-dgtw2-lab2`（4カメラ: cam01, cam02, cam03, cam04。1,800フレーム/カメラ、30fps、1920x1080、単一人物）
- HALPE_26スケルトン（26キーポイント）
- 追加ライブラリのインストールは行わない
